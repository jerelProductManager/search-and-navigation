What Are Nested Refinements?

Rule-Based Refinements

Nested Refinements are a type of conditional rule-based refinement — one of two rule-based types alongside application-based refinements.

They introduce a parent-child grouping layer on top of existing spec refinements, enabling structured navigation of large refinement sets.

Application-Based

Refinements driven by minimum and maximum requirements set for spec refinements. If an item meets these standards it becomes part of the result set for that application-based refinement category.

Conditional (Nested Refinements) ←

Dynamic parent-child groupings driven by shared characteristics within a spec refinement's values. Smart, scalable, context-aware.

Problems We're Solving

1

Poor Scannability

When 100+ refinement values are paginated across multiple pages, customers struggle to form a mental model of available options. They can't quickly assess scope or know what they're selecting against.

2

Pagination Friction

The modal-based "See All" experience forces users to page through long lists. This breaks flow when they're trying to narrow to a precise set of values — every page turn is friction against their goal.

3

Select All & Search Conflicts

"Select All" and search within the refinement interact poorly with pagination — customers think they've selected everything when they haven't, creating confusion and incorrect filtered results.

How It Works

1

Spec Refinement

Start with a spec refinement containing a large set of values (e.g. Native Resolution with 100 values).

2

Identify Shared Traits

Detect subsets with shared characteristics — e.g. resolutions that are 2K, 4K, 6K, or 8K.

3

Create Parent Groups

Define conditional parent refinements (2K, 4K, 6K, 8K) that represent those clusters.

4

Nest Spec Values

The original spec refinement values nest under their matching parent. Browsing becomes hierarchical and scannable.

Example: Native Resolution spec has 100 values → Parents created: 2K · 4K · 6K · 8K → Each resolution value nests under its parent group.

Where It Appears

Refinement Panel (Search & Browse)

Customers filtering product listings encounter conditional refinements in the left-rail or inline filter UI. Parent groups are shown first; expanding them reveals spec-level values. The drawer replaces the modal for full-list browsing.

Owned by: Search & Requirements — Yehuda's domain

Nav Bar — Rule Configuration (BHIA)

The Nav Arc Team accesses a configuration UI within the Nav Bar to create and manage conditional parent refinements and define their relationships to spec refinements.

Owned by: Nav Bar / Internal Tools

The Solution

Modal → Drawer (See All)

Replace the modal-based "See All" experience with a drawer. The drawer eliminates pagination entirely — users scroll vertically through all groups and orphan values. Select All and search state are consistent regardless of list length. This ships first and can run concurrently with the Nav Bar ticket.

Conditional Parent Groupings

With parent refinements created in the Nav Bar (BHIA ticket), shared-characteristic values are grouped under scannable headers. The spec refinement nests underneath, preserving full detail without overwhelming the customer. The Nav Arc Team defines the rules — the system handles the hierarchy.

✓  UX design is complete and approved. Front-end execution is tracked via BHJ ticket.
